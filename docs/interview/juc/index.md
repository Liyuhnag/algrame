# JUC 并发编程

这一部分主要整理 Java 并发编程相关内容，核心包括线程基础、锁机制、CAS、线程池、并发容器以及线程协作工具。

### Java 中如何创建多线程？

Java 中创建多线程，常见的方式有四种：继承 `Thread`、实现 `Runnable`、实现 `Callable` 配合 `FutureTask`，以及通过线程池创建和管理线程。面试时不建议只背结论，最好顺便说明每种方式的适用场景和限制。

面试时可以这样回答：

::: tip 标准回答
Java 中创建多线程，常见有四种方式。

第一种是继承 `Thread` 类，重写 `run()` 方法，然后调用 `start()` 启动线程。这种方式最直观，但因为 Java 是单继承，如果业务类已经有父类，就不方便再继承 `Thread`，所以实际开发中一般不推荐作为首选。

第二种是实现 `Runnable` 接口，重写 `run()` 方法，再把任务交给 `Thread` 去执行。这种方式把“任务”和“线程”解耦了，扩展性更好，也是传统开发里更常见的写法。它适合不需要返回结果、也不能直接抛出受检异常的场景。

第三种是实现 `Callable` 接口，并配合 `FutureTask` 或线程池提交执行。`Callable` 和 `Runnable` 最大的区别在于，`call()` 方法可以有返回值，也可以抛出异常，所以更适合需要获取异步执行结果的场景。

第四种是通过线程池来创建和管理线程，比如 `ExecutorService`。这也是生产环境里最推荐的方式。因为线程的创建和销毁成本比较高，如果任务会被频繁执行，直接反复创建线程会带来明显的性能损耗，而线程池可以复用线程、控制并发数量、统一管理任务生命周期。

如果只从工程实践角度回答，一般可以补一句：单纯从“能不能创建线程”来看，前面三种都可以；但在真实项目里，通常更推荐使用线程池。
:::

::: tip 回答提醒
`Callable` 适合的是“需要返回值、允许抛异常”的场景，不是“不需要返回值”。这点面试里很容易说反。
:::

### 什么是协程？Java 支持协程吗？

协程可以理解为一种比线程更轻量级的执行单元，它通常在用户态完成调度，能够在执行过程中主动挂起和恢复，从而以较低的切换成本实现高并发。和线程相比，协程更强调“协作式调度”和轻量级切换。

面试时可以这样回答：

::: tip 标准回答
协程本质上是一种用户态的轻量级并发模型。它和线程最大的区别在于，线程一般由操作系统调度，而协程通常由应用程序运行时在用户态调度，所以切换成本更低，不需要频繁陷入内核态。

在 Java 里，如果说“是否原生支持协程”，更严谨的回答是：Java 语言本身长期没有提供像 Go、Kotlin 那样的协程语法和协程模型。JDK 21 引入了虚拟线程，这是 Java 并发模型的一次重要增强，但虚拟线程并不等同于传统意义上的协程。

虚拟线程可以理解为由 JVM 管理的轻量级线程。它最大的价值是把“一任务一线程”的编程方式重新变得可行，让开发者仍然使用阻塞式代码风格，但底层由 JVM 负责调度和复用少量平台线程。

它和传统平台线程的主要区别有两点。

第一，映射关系不同。传统线程基本上是 1:1 映射到操作系统内核线程，创建、销毁和切换成本都比较高。虚拟线程则由 JVM 调度到少量平台线程上执行，属于更轻量的实现方式，因此能够支撑远高于传统线程数量级的并发任务。

第二，阻塞代价不同。传统线程一旦阻塞，往往会直接占住底层内核线程。虚拟线程在大多数 JDK 可感知的阻塞操作中，会被 JVM 挂起并从底层载体线程上卸载，这样 Carrier Thread 可以继续执行其他虚拟线程，从而显著提升资源利用率。

不过要注意，虚拟线程不是协程语法，也不是完全等价于 Kotlin Coroutine。面试时最好表述成：Java 在 JDK 21 通过虚拟线程提供了更轻量级的并发模型，但严格来说，这不是语言层面的协程支持。
:::

::: tip 高频追问
如果面试官继续问 Java 为什么没有严格意义上的协程，可以补一句：Java 更偏向在线程模型上演进，Project Loom 选择的是“保留线程编程模型，但把线程做轻”，这样兼容性更好，业务代码迁移成本也更低。
:::

### Java 中的线程安全是什么意思？

线程安全是指多个线程同时访问同一段代码或同一份共享数据时，不管线程如何调度，程序都能够表现出正确且符合预期的结果，不会出现数据不一致、状态错乱或者业务逻辑错误。

面试时可以这样回答：

::: tip 标准回答
Java 中的线程安全，本质上是指在并发环境下，程序对共享资源的访问仍然是正确、稳定、可预期的。换句话说，多线程同时执行时，结果不能因为执行顺序不同就出错。

线程不安全问题通常来自三个方面。

第一是可见性问题。一个线程修改了共享变量，另一个线程不一定能立刻看到最新值，因为线程可能会先操作自己的工作内存，再在某个时机同步到主内存。

第二是原子性问题。一个看起来是单步的操作，实际底层可能由多步组成，比如 `i++` 本质上包含读取、计算、写回三个步骤，中间如果被其他线程打断，就可能得到错误结果。

第三是有序性问题。编译器和 CPU 为了优化性能，可能会对指令进行重排序。单线程场景下通常没有问题，但在多线程场景下，其他线程可能观察到一个不符合直觉的执行顺序。

常见的保证线程安全的方式主要有几类。第一类是加锁，比如 `synchronized` 和 `ReentrantLock`，适合保护临界区代码。第二类是使用原子类，比如 `AtomicInteger`，通过 CAS 保证单个变量更新的原子性。第三类是使用并发容器，比如 `ConcurrentHashMap`，把常见的并发访问场景封装好。第四类是通过 `ThreadLocal` 做线程隔离，让每个线程拥有自己的独立变量副本，从而避免共享带来的并发问题。
:::

::: tip 回答提醒
`ThreadLocal` 更准确地说是“避免共享”，而不是直接解决多个线程竞争同一份数据的问题。面试时这样区分，表达会更严谨。
:::

### 什么是 Java 中的线程同步？有哪些实现方式？

线程同步是指在多线程环境下，通过一定的机制控制多个线程对共享资源的访问顺序和访问方式，保证结果的正确性，避免出现竞态条件、数据不一致等问题。它的核心不仅是“同一时刻只允许一个线程进入临界区”，还包括线程之间的协作与执行时机控制。

面试时可以这样回答：

::: tip 标准回答
Java 中的线程同步，本质上是为了保证共享资源在并发访问场景下的正确性，核心目标包括原子性、可见性和有序性。常见的实现方式主要有三类。

第一类是 `synchronized`。它是 Java 提供的内置锁，可以直接修饰实例方法、静态方法或者代码块，由 JVM 帮我们完成加锁和释放锁，写法简单，而且是可重入的，适合大多数基础同步场景。

第二类是 `Lock` 接口及其实现类，比如 `ReentrantLock`。它需要手动调用 `lock()` 和 `unlock()`，通常会配合 `try-finally` 使用。相比 `synchronized`，它的灵活性更高，支持可中断锁、超时获取锁、公平锁以及多个 `Condition` 条件队列，适合更复杂的并发控制。

第三类是 JUC 提供的原子类和并发同步工具。原子类如 `AtomicInteger`、`AtomicLong`，底层基于 CAS，适合做单个共享变量的无锁原子更新。并发同步工具如 `CountDownLatch`、`CyclicBarrier`、`Semaphore`，更适合做线程协作控制，比如等待一组线程执行完成、让多个线程同时开始，或者限制同一时刻可访问某资源的线程数量。

需要注意的是，`CountDownLatch`、`CyclicBarrier`、`Semaphore` 严格来说不是原子类，而是并发同步工具。
:::

::: tip 高频追问
如果面试官继续问 `synchronized` 和 `ReentrantLock` 的区别，可以重点回答两点：一是 `synchronized` 由 JVM 自动释放锁，使用更简单；二是 `ReentrantLock` 功能更丰富，适合需要可中断、超时控制或公平锁的场景。
:::

### Java 中线程之间如何进行通信？

Java 中线程通信的本质，是让多个线程在并发执行时能够协调配合，知道什么时候该等待、什么时候该继续执行、什么时候可以消费或生产数据。常见思路可以概括为两类：共享内存和显式通知。

面试时可以这样回答：

::: tip 标准回答
Java 线程之间通信，本质上是多个线程围绕共享状态进行协作。最底层的方式是共享内存，也就是多个线程读写同一份数据；而更常见的工程化方式，是在共享状态的基础上配合等待、唤醒或阻塞队列来完成线程之间的协调。

如果从原理上看，共享内存模型下要重点解决三个问题：可见性、原子性和有序性。比如一个线程改了标志位，其他线程要能及时看到；多个线程共同修改一个计数器时，不能互相覆盖；代码执行顺序也不能因为重排序而让其他线程看到异常状态。

从实现角度看，最常见的线程通信方式主要有四类。

第一类是 `wait()` / `notify()` / `notifyAll()` 机制。它们必须配合 `synchronized` 使用。线程在条件不满足时调用 `wait()` 进入等待，同时释放锁；其他线程在条件满足后调用 `notify()` 或 `notifyAll()` 唤醒等待线程。这是最经典的线程协作方式。

第二类是 `Lock` + `Condition`。比如 `ReentrantLock` 可以通过 `newCondition()` 创建多个条件队列。相比 `wait/notify`，它的好处是一个锁可以配多个等待条件，唤醒粒度更细，写复杂协作逻辑时更灵活。

第三类是 `BlockingQueue`。这是生产者消费者模型里最实用的一种方案。生产者调用 `put()`，队列满了会阻塞；消费者调用 `take()`，队列空了会阻塞。很多场景下直接用阻塞队列比手写 `wait/notify` 更安全也更简洁。
