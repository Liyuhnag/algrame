# 单一职责原则

单一职责原则的英文是 `Single Responsibility Principle`，简称 `SRP`。它的核心思想可以概括为一句话：一个类或模块，应该只负责一类职责，或者说只承担一个变化原因。

很多人第一次接触这个原则时，会把它理解成“一个类只能有一个方法”或者“类拆得越细越好”。这两种理解都不准确。单一职责原则真正想解决的是：**避免把无关职责耦合在一起，降低修改影响面，提高代码的可读性、可维护性和可扩展性。**

## 什么是单一职责原则

单一职责原则强调，不要设计大而全的类，而要设计职责明确、边界清晰、功能相对单一的类。

换句话说，如果一个类里混杂了两个甚至多个不相关的业务功能，那么这个类通常就不够“单一”，后续会出现以下问题：

- 修改一个职责时，容易影响另一个职责
- 类越来越大，阅读和维护成本越来越高
- 测试难度增加，复用性变差
- 命名会越来越抽象，只能用 `Manager`、`Context` 这类模糊名字兜底

举个最简单的例子：一个类里既处理订单逻辑，又处理用户逻辑。订单和用户本来就是两个相对独立的业务概念，把它们强行放在一个类里，就违反了单一职责原则。更合理的做法是拆成订单类和用户类。

## 如何判断职责是否单一

单一职责原则看起来简单，但在真实项目里，“一个类是否只做了一件事”并没有绝对标准。很多时候，它取决于业务背景、阶段目标和拆分粒度。

### 示例：`UserInfo` 是否满足单一职责原则

假设在一个社交产品中，我们用下面这个类记录用户信息：

```java
public class UserInfo {
    private long userId;
    private String username;
    private String email;
    private String telephone;
    private long createTime;
    private long lastLoginTime;
    private String avatarUrl;
    private String provinceOfAddress;
    private String cityOfAddress;
    private String regionOfAddress;
    private String detailedAddress;
    // ...省略其他属性和方法...
}
```

这个类是否满足单一职责原则，通常会有两种看法：

- 第一种观点认为，它保存的都是“用户相关信息”，从业务主语上看仍然属于同一职责，因此是合理的。
- 第二种观点认为，地址信息已经形成了相对独立的数据块，未来很可能和物流、收货、订单等场景耦合，应该拆分成独立的 `UserAddress`。

这两种观点都不一定错，关键还是看业务场景。

- 如果地址信息只是个人主页展示的一部分，那么当前设计完全可以接受。
- 如果后续引入电商、物流、收货地址管理等场景，那么把地址信息从 `UserInfo` 中拆出来，会更合理。

如果业务再继续扩展，比如公司内部要做统一账号系统，那么 `email`、`telephone`、登录认证信息也可能继续拆到更独立的认证模型中。

这说明一件事：**职责是否单一，不是静态不变的，它会随着业务复杂度、领域边界和系统演进而变化。**

## 更实用的判断信号

在工程实践里，与其抽象地争论“这个类算不算单一”，不如看一些更可执行的信号：

- 类中的代码行数、方法数、属性数明显过多，已经影响理解和维护
- 类依赖了很多外部类，或者反过来有很多类强依赖它，耦合度过高
- 很难给类起一个准确的名字，只能用模糊词汇命名
- 类中不同方法明显围绕不同属性组工作，像是多块逻辑硬塞在一起
- 某一部分字段和行为频繁被单独修改，已经形成了独立演进节奏

如果这些现象开始变得明显，就说明这个类很可能该拆了。

## 单一职责原则不是越细越好

很多人学到这里，会自然得出一个结论：既然职责越单一越好，那是不是应该把类拆得越细越好？答案是否定的。

我们看一个简单的序列化例子：

```java
/**
 * Protocol format: identifier-string;{gson string}
 * For example: UEUEUE;{"a":"A","b":"B"}
 */
public class Serialization {
    private static final String IDENTIFIER_STRING = "UEUEUE;";
    private Gson gson;

    public Serialization() {
        this.gson = new Gson();
    }

    public String serialize(Map<String, String> object) {
        StringBuilder textBuilder = new StringBuilder();
        textBuilder.append(IDENTIFIER_STRING);
        textBuilder.append(gson.toJson(object));
        return textBuilder.toString();
    }

    public Map<String, String> deserialize(String text) {
        if (!text.startsWith(IDENTIFIER_STRING)) {
            return Collections.emptyMap();
        }
        String gsonStr = text.substring(IDENTIFIER_STRING.length());
        return gson.fromJson(gsonStr, Map.class);
    }
}
```

如果硬要追求“更单一”，可以把它拆成一个 `Serializer` 和一个 `Deserializer`：

```java
public class Serializer {
    private static final String IDENTIFIER_STRING = "UEUEUE;";
    private Gson gson;

    public Serializer() {
        this.gson = new Gson();
    }

    public String serialize(Map<String, String> object) {
        StringBuilder textBuilder = new StringBuilder();
        textBuilder.append(IDENTIFIER_STRING);
        textBuilder.append(gson.toJson(object));
        return textBuilder.toString();
    }
}

public class Deserializer {
    private static final String IDENTIFIER_STRING = "UEUEUE;";
    private Gson gson;

    public Deserializer() {
        this.gson = new Gson();
    }

    public Map<String, String> deserialize(String text) {
        if (!text.startsWith(IDENTIFIER_STRING)) {
            return Collections.emptyMap();
        }
        String gsonStr = text.substring(IDENTIFIER_STRING.length());
        return gson.fromJson(gsonStr, Map.class);
    }
}
```

表面上看，拆分后的职责更单一了，但问题也出现了：

- 协议常量和协议规则重复出现在两个类里
- 一旦协议格式变化，需要同时修改两个类
- 如果只改了其中一个类，很容易出现序列化和反序列化不匹配

这说明，**过度拆分会破坏内聚性，增加维护成本，并不一定比原来的设计更好。**

## 实践中的正确姿势

在实际开发中，更推荐采用下面这套思路：

1. 先用满足当前业务的合理粒度建模，不要一开始就过度拆分。
2. 随着业务演进，观察类是否出现“职责膨胀”。
3. 一旦某部分字段、方法、依赖关系已经形成独立变化趋势，再逐步拆分。
4. 拆分的最终目标，不是形式上的“单一”，而是整体代码的可读性、可维护性和可扩展性更好。

这也是为什么，单一职责原则往往和“持续重构”联系在一起。很多时候，我们不是一开始就设计出最优结构，而是在系统演进过程中不断把职责边界调整得更清晰。

## 总结

单一职责原则的核心，不是机械地把类拆小，而是让每个类的职责边界清晰、变化原因单一。

你可以把它记成一句面试里的总结话术：

::: tip 面试回答
